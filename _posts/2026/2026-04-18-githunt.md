I started playing around with LinkedIn’s search and hiring tools recently.

They’re powerful, but also surprisingly expensive once you go beyond basic usage. And the more I used them, the more a second issue stood out: they rely heavily on self-declared skills.

Which makes sense, but also creates a problem.

It is very easy to list SQL, Python, dashboards and analytics experience. It is much harder to actually show what that looks like in practice.

For early-career candidates especially, profiles can end up looking very similar. The signal is there, but it is buried under formatting, phrasing and how well someone knows how to present themselves.

For technical roles, we can use a technical approach...

This is a small proof of concept pipeline that pulls public GitHub activity, extracts real code, and uses an LLM to classify what someone can actually do.

The goal is not to automate hiring. It is to build a better shortlist using evidence instead of self-description.

## Step 1: Find Melbourne / Victoria candidates

The first script just brute-forces GitHub user search with location queries like

- Melbourne  
- Melbourne Australia  
- Melbourne VIC  
etc

Filtered to accounts created after 1 Jan 2023 so it leans towards newer / early-career profiles.

For each user it pulls:

- profile  
- repos  
- recent activity (last ~6 months)  

## Step 2: Pull actual code (not repo names)

This is where it becomes useful.

Instead of reading entire repositories (slow and mostly irrelevant), the second script:

1. Picks a few relevant repos per candidate  
2. Walks recent commits  
3. Pulls only recent `.sql` and `.py` files  

Current setup:

- up to 4 repos per candidate  
- up to 10 recent SQL files  
- up to 10 recent Python files  

The “recent commits” bit matters. You want what they touched last week, not a notebook from 2022.

Rough shape:

```python

MAX_RECENT_PY_FILES_PER_REPO = 10

def get_recent_sql_and_py_files_from_commits(...):
    # scan recent commits
    # collect modified .sql and .py paths
    # stop once limits are hit
    return sql_paths + py_paths
```

Files are trimmed and bundled into a single “code packet” per candidate.

This keeps things:

- small enough to process  
- focused on real work  
- not dominated by junk files / exports  

## Step 3: Get a structured read on the code

Each code packet gets sent to an LLM with a constrained prompt.

Not “is this person good”.

Just:

- can they write SQL  
- can they write Python  
- do they structure things properly  
- is there any actual analytical thinking  

Prompt is intentionally narrow:

```text
You are reviewing public GitHub code for evidence of early-career analytics skill.

Assess:
- SQL skill
- Python skill
- SQL structure and readability
- SQL/Python interoperability
- Segmentation / analytical thinking

Rules:
- Use only the provided code
- Be conservative
- Cite actual file/path evidence
- Return strict JSON only
```

Output goes to:

- `candidate_code_reviews.csv`

So you end up with something sortable:

- SQL score  
- Python score  
- structure score  
- segmentation score  
- evidence summary  
- strengths / gaps / red flags  

Already more useful than most resumes.

## What this is actually doing (and not doing)

This is not ranking “the best candidates”.

It is filtering for people who have:

- written non-trivial SQL  
- written non-trivial Python  
- structured something end-to-end  
- touched code recently  

That alone removes a large chunk of noise.

It is also very good at surfacing people who:

- don’t optimise their LinkedIn  
- don’t keyword-stuff resumes  
- just quietly build things  

Which is usually a useful group.

## What it misses

Plenty:

- good candidates with private repos → invisible  
- weak GitHub profiles → undersold  
- partial repo sampling → incomplete view  
- API limits → slow  

Also: GitHub activity is not evenly distributed. Some people just don’t work in public.

So this has bias. It is just a different bias to resume screening.

## Practical note

Resumability ended up being essential.

Everything writes to CSV + JSON state so the scripts can stop and start without losing progress.

Because once you start pulling commits across multiple repos, you will hit rate limits or just run out of patience.

## Next improvements

- cache repo trees + file contents  
- better repo selection before deep scan  
- stronger filtering before LLM step  
- better prompts for actual analytics thinking  
- handle notebooks properly  

Also need to check whether the rankings match real hiring outcomes.

Otherwise this just becomes a very efficient way to automate my own bias.
