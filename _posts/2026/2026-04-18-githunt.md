My ten minute impression of LinkedIn recruitment tools was that they are expensive and limited by the nature of the platform. They rely on people to self-declare their skills and experience.

It is easy to list SQL, Python, dashboards and analytics experience. It is much harder to see what that actually looks like in practice.

For early-career candidates in particular, profiles can end up looking very similar. High signal to noise, with any real signal lost in buzzwords, AI-generated bios and humble-brags.

For technical roles, there is a more direct option.

You can look at the code.

Instead of relying on profiles, you can assess people based on their actual projects and repositories.

Rather than waiting for candidates to apply, you start from public work and work backwards to the person.

This shows people in a more natural state. Not a polished profile, but the way they actually write queries, structure scripts, and organise their work.

It also surfaces who is spending time building things outside of work or study: small projects, half-finished ideas, incremental commits.

This is a small proof-of-concept pipeline that pulls public GitHub activity, extracts real code, and uses an LLM to assess and classify what someone can actually do.

The goal is not to automate hiring. It is just a way to build a better shortlist using actual work rather than self-description.


## Step 1: Find Melbourne / Victoria candidates

The first script just brute-forces GitHub user search with location queries like

- Melbourne  
- Melbourne Australia  
- Melbourne VIC  
etc

Filtered to accounts created after 1 Jan 2023 so it leans towards newer / early-career profiles.

## Step 2: Use an LLM to assess real code

Once you have a small packet of recent SQL and Python code, you can actually evaluate what someone can do.

Each candidate gets a bundled “code packet” (recent `.sql` and `.py` files pulled from their commits), which is sent to an LLM with a constrained prompt.

The goal is not “is this person good”.

It’s much narrower:

- can they write SQL  
- can they write Python  
- do they structure queries and logic properly  
- is there any evidence of analytical thinking  

The prompt is intentionally strict:

```text
You are reviewing public GitHub code for evidence of early-career analytics skill.

Assess:
- SQL skill
- Python skill
- SQL structure and readability
- SQL/Python interoperability
- Segmentation / analytical thinking

Rules:
- Use only the provided code
- Be conservative
- Cite actual file/path evidence
- Return strict JSON only

```
