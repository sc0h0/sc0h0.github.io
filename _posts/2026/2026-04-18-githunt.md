Hiring junior analytics talent is awkward.

At the junior end of the market, resumes often tell you very little. Plenty of candidates list SQL, Python, dashboards and analytics projects, but it is hard to know who has actually built anything versus who has simply completed a course or copied a project.

So I started building a small GitHub-based pipeline to find Melbourne and regional Victoria candidates using public evidence of coding activity.

The idea was not to replace normal hiring processes, but to build a better shortlist using real projects as an input.

## Step 1: Find Melbourne / Victoria candidates

The first script searches GitHub users using location queries such as:

- Melbourne
- Melbourne Australia
- Melbourne VIC
- Geelong
- Ballarat
- Bendigo
- Warrnambool
- Shepparton

It only looks at accounts created after 1 January 2023, writes discovered users to a CSV, and keeps track of progress so the run can resume if interrupted.

For each discovered user, the script then fetches:

- their profile
- their repos
- recent personal repo activity over the last 6 months

It uses a lightweight set of inclusion rules based on signals like:

- Melbourne / Victoria location indicators
- Victorian university mentions in bio or company
- student / graduate wording
- data-related keywords such as SQL, Python, Tableau, Power BI, pandas, Jupyter, and analytics terms

This stage is intentionally simple. It is designed to gather a candidate pool, not make a final judgement.

The outputs are:

- `melbourne_users.csv` — all discovered users plus status tracking
- `melbourne_candidates.csv` — shortlisted candidates from the lightweight scan
- `simple_01_state.json` — state file so the run can resume without starting again

## Step 2: Pull code from the most relevant repos

The second script takes shortlisted candidates and performs a deeper review.

Rather than trying to read every file in every repository, it selects up to a small number of relevant personal repos per candidate, based on repo names, descriptions, languages and recent activity.

It then walks the repo tree and selectively pulls:

- `.sql`
- `.py`
- `.ipynb`
- selected README / config files

It also skips large or irrelevant files such as images, exports, build artefacts and other non-code assets.

This keeps the review focused and limits API usage.

The outputs from this stage are assembled into a structured “code packet” for each candidate.

## Step 3: Use an LLM to assess technical signals

Those code packets are then sent to the Anthropic API with a structured prompt asking for a conservative assessment of:

- SQL skill
- Python skill
- commenting and documentation quality
- how well SQL is broken into logical chunks
- SQL/Python interoperability
- segmentation or analytical thinking
- any collections / debt recovery evidence

The result is written to:

- `candidate_code_reviews.csv`

This file contains structured fields such as:

- SQL score
- Python score
- commenting quality
- SQL structuring score
- segmentation score
- collections domain score
- evidence summary
- strengths, gaps and red flags

## What I like about this approach

The point is not that GitHub gives you the full truth.

The point is that public projects often reveal things a resume cannot:

- whether someone can structure a query cleanly
- whether they can connect Python and SQL in a sensible workflow
- whether they comment their work properly
- whether they think analytically or just throw code together

For junior candidates, this is often more informative than a polished CV.

## Limitations

There are a few obvious limitations.

First, this is heavily constrained by GitHub API rate limits, especially during the deeper repo review step.

Second, many good candidates have incomplete public profiles, sparse bios, or private work that never appears on GitHub.

Third, the second-stage review is only looking at selected files from selected repos, not every line of code in a candidate’s account.

So this is best thought of as a ranking and discovery tool, not an automated hiring decision-maker.

## Resume support

Both stages were written to resume from where they left off, using CSV outputs and JSON state files to avoid rework.

That turned out to be essential, because the deeper repo inspection step quickly runs into API limits and can take a long time to complete.

## Next improvements

The next things I want to improve are:

- better caching of repo trees and file contents
- stronger filtering before deep repo inspection
- more efficient handling of GitHub rate limits
- improved prompts for analytics-domain signals like segmentation and collections strategy
