Hiring junior data analysts is surprisingly inefficient.

Most candidates look similar on paper, resumes are often inflated, and it's hard to tell who actually *builds things* versus who just lists tools.

So I decided to build my own pipeline to find candidates based on what they’ve actually created — using GitHub as the source of truth.

The idea was simple:
- Find people in Melbourne / Victoria
- Identify students, recent grads, or hobby coders
- Prioritise those actively building projects (SQL, Python, dashboards, etc.)

And most importantly:
**use their actual code to assess skill level, not just their resume**

---

## Step 1: Finding Melbourne candidates

The first step was to cast a wide net.

I used the GitHub API to search for users with location signals like:
- "Melbourne"
- "Melbourne VIC"
- "Geelong", "Ballarat", etc.

Then I filtered them based on:
- recent activity (last 6 months)
- personal repos (not forks)
- signals of data work (SQL, Python, analytics)

This produced a candidate pool of ~2,000 users.

From there, I applied some simple logic to classify candidates into buckets:

**1. Local junior coder**
- Melbourne/Victoria signal
- some personal coding activity
- relatively new GitHub account

**2. Victorian student / recent grad**
- university or coursework signals
- repo names like "assignment", "capstone", etc.
- may or may not list location explicitly

**3. Local hobby coder**
- Melbourne/Victoria signal
- multiple personal projects
- no obvious uni signals

This stage is intentionally broad — the goal is recall, not precision.

---

## Step 2: Filtering for signal (cheap pass)

Before doing anything expensive, I filter candidates using lightweight heuristics:

- Do they have recent activity?
- Do they have personal repos?
- Any data-related signals? (SQL, Python, dashboards)
- Any uni / grad indicators?

This reduces the pool significantly while keeping interesting profiles.

At this stage, I’m not trying to be perfect — just removing obvious noise.

---

## Step 3: Extracting actual code

This is where things get interesting.

For each candidate, I:
- select their most relevant repos (Python, SQL, analytics-heavy)
- scan repo structure
- pull:
  - `.sql` files
  - `.py` files
  - Jupyter notebooks
  - READMEs

Then I build a structured “code packet” containing:
- repo names
- file paths
- actual code snippets

This gets fed into an LLM for analysis.

---

## Step 4: LLM-based skill assessment

Instead of guessing based on keywords, I evaluate candidates based on their *actual code*.

The model scores things like:

- SQL skill (joins, CTEs, structure)
- Python skill (data manipulation, libraries)
- commenting / documentation quality
- SQL → Python workflows
- segmentation / analytical thinking
- any exposure to collections / debt recovery logic

Output looks something like:

- SQL skill: 7/10
- Python: 6/10
- Segmentation: strong
- Collections domain: none
- Overall: good junior analyst potential

This is where the pipeline becomes genuinely useful.

---

## Why this works better than resumes

A resume tells you what someone *claims*.

GitHub shows you:
- how they structure problems
- how they write queries
- whether they actually finish projects
- how they think

You can quickly spot:
- tutorial clones vs original work
- messy vs structured SQL
- real analytical thinking vs surface-level work

---

## Challenges

This wasn’t as straightforward as it sounds.

### 1. GitHub rate limits
Pulling repo trees and file contents is expensive:
- ~30–100 API calls per candidate
- easy to hit limits quickly

### 2. Incomplete profiles
Many users:
- don’t list location
- don’t mention university
- have minimal bios

So the system needs to work with partial signals.

### 3. Signal vs noise
Not all repos are useful:
- forks
- tutorials
- empty projects

Filtering meaningful work is non-trivial.

---

## What I’d improve next

- Better pre-filtering before deep code extraction
- Caching repo structures to reduce API usage
- More targeted repo selection (avoid noise)
- Stronger prompts for domain-specific detection (e.g. collections strategies)

---

## Final thoughts

This approach flips the hiring process:

Instead of:
> resume → interview → test

You start with:
> real work → skill signal → shortlist

It’s not perfect, but it’s a much stronger signal for junior candidates — especially those without traditional experience.

And the best part:
you uncover people who wouldn’t even apply.
