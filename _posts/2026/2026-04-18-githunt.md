LinkedIn has tools to identify specific skill sets however they're only what is written and not what is actually a skill. It's also expensive and likely biased to the most jacked up profiles. 

Seeking technical roles means that we can be more ojective in what we identify as a skill, and the following shows an proof of concept approach to glean personal GitHub repositories and put them through LLMs to classify and cateogories skills so that I can contact them for employment conversations. 

## Step 1: Find Melbourne / Victoria candidates

The first script just brute-forces GitHub user search with location queries:

- Melbourne  
- Melbourne Australia  
- Melbourne VIC  
- Geelong  
- Ballarat  
- Bendigo  
- Warrnambool  
- Shepparton  

Filtered to accounts created after 1 Jan 2023 so it leans towards newer / junior profiles.

For each user it pulls:

- profile  
- repos  
- recent activity (last ~6 months)  

Then runs a very simple set of signals:

- location looks like VIC  
- mentions a Victorian uni  
- has “student”, “grad”, etc  
- mentions SQL / Python / analytics keywords  
- has actually pushed something recently  

That’s it.

No scoring model. No cleverness. Just filtering out obvious non-candidates.

Outputs:

- `melbourne_users.csv` — everything found  
- `melbourne_candidates.csv` — filtered list  
- `simple_01_state.json` — so it can resume after rate limits  

The point of this stage is just to avoid manually opening hundreds of profiles.

## Step 2: Pull actual code (not repo names)

This is where it becomes useful.

Instead of reading entire repositories (slow and mostly irrelevant), the second script:

1. Picks a few relevant repos per candidate  
2. Walks recent commits  
3. Pulls only recent `.sql` and `.py` files  

Current setup:

- up to 4 repos per candidate  
- up to 10 recent SQL files  
- up to 10 recent Python files  

The “recent commits” bit matters. You want what they touched last week, not a notebook from 2022.

```python
    # stop once limits are hit

```

Files are trimmed and bundled into a single “code packet” per candidate.

This keeps things:

- small enough to process  
- focused on real work  
- not dominated by junk files / exports  

## Step 3: Get a structured read on the code

Each code packet gets sent to an LLM with a constrained prompt.

Not “is this person good”.

Just:

- can they write SQL  
- can they write Python  
- do they structure things properly  
- is there any actual analytical thinking  

Prompt is intentionally narrow:

```text
You are reviewing public GitHub code for evidence of junior analytics skill.

Assess:
- SQL skill
- Python skill
- SQL structure and readability
- SQL/Python interoperability
- Segmentation / analytical thinking

Rules:
- Use only the provided code
- Be conservative
- Cite actual file/path evidence
- Return strict JSON only
```

Output goes to:

- `candidate_code_reviews.csv`

So you end up with something sortable:

- SQL score  
- Python score  
- structure score  
- segmentation score  
- evidence summary  
- strengths / gaps / red flags  

Already more useful than most resumes.

## What this is actually good for

This is not a hiring system.

It is a **shortlisting tool**.

It works well for:

- finding people who actually build things  
- filtering out empty profiles quickly  
- seeing how someone thinks in code  

For junior candidates, that signal is usually stronger than anything written on a CV.

## What it misses

Plenty:

- good candidates with private repos → invisible  
- weak GitHub profiles → undersold  
- partial repo sampling → incomplete view  
- API limits → slow  

Also: GitHub activity is not evenly distributed. Some people just don’t work in public.

So this has bias. It is just a different bias to resume screening.

## Practical note

Resumability ended up being essential.

Everything writes to CSV + JSON state so the scripts can stop and start without losing progress.

Because once you start pulling commits across multiple repos, you will hit rate limits.
